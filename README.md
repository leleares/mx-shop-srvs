### 分布式锁
锁是操作系统给予的资源调度的能力，对同一台服务器上的并发请求使用锁可以很好的解决资源竞争问题。但是一旦统一服务部署为多台服务器，各个服务器之间都有各自的一把锁，无法对同一资源进行很好的并发资源控制，分布式锁就是解决这个问题的，可以理解为抽象于各个服务器之上的一把锁，各个服务共享。

常见的分布式锁实现方案
- 基于Mysql的悲观锁、乐观锁
- 基于redis的分布式锁
- 基于zookeeper的分布式锁

MySql悲观锁：
我很悲观，总是认为数据资源会有人来抢，所以趁早把资源锁住。（MySql语法支持）
注意：悲观锁在MySql中属于行锁，即会锁住一条记录，但前提是for update的查询条件字段是有索引的，如果没有索引，就不是锁一条记录了，而是会升级成表锁。
```sql
START TRANSACTION;

SELECT stock
FROM inventory
WHERE good_id = 1 //注意good_id一定为加了索引的字段
FOR UPDATE; // 这里就把 good_id = 1的这条记录上锁了

UPDATE inventory
SET stock = stock - 5
WHERE good_id = 1; 

COMMIT; // 释放锁
```

我很乐观，等真正提交时再检查有没有冲突。（通过加字段来模拟实现）

### 基于redis的分布式锁
流程：连接redis获取redis client，通过redis client创建连接池pool，使用第三方库redsync基于pool创建mutex，在竞争资源操作前lock，操作完unlock。
原理：拿商品421举例子，当对商品421进行加锁时，会在redis中创建一个标志位，假设为421设置为1，当另一个进程也想对商品421进行操作时，通过查询redis发现该商品已经被锁上，则等到锁的释放才能操作。
核心原理1:【setnx】｜对商品进行加锁时，本质上是两步操作，第一步是查询421在redis中的状态（get），第二步才是设置421在redis中的状态（set），既然是两步操作，高并发场景下就容易被中断，导致锁机制混乱，因此需要一个原子操作，这就是redis的setnx，该操作直接对421进行设置，如果421不存在则设置成功，否则设置失败。
核心原理2:【延时】｜ 想象这样一个场景，如果某个线程获取到锁之后，在释放锁之前挂机了，那么这个锁将无法释放，那么其他线程将无法拿到资源操作的权力，解决方案就是给421这个redis的key加一个过期时间假设为8s，但是要是我正常的业务逻辑就执行了9s，那就是在我正常逻辑执行结束前锁就释放了，别人就拿到了，那锁机制又混乱了，解法就是在正常逻辑结束前去进行刷新过期时间，这个redsync库支持。不过仍然有问题，假想某个进程有bug，不会终止，那她一直刷新redis的key的过期时间别人永远也别想拿到资源的权力。
核心原理3:【安全性】 ｜ 锁只能被锁住的人解开，这是通过redis的key里面的value实现的，设置锁的人会往421这个key中写入一个唯一的value，解锁时再拿这个值去跟redis中的key对应的value做对比，如果是就能解开。
核心原理4:【redlock 红锁】 ｜ redis服务通常不是单一节点，通常是集群，那就会有一个master节点和多个slave节点，正常情况下，master节点会确保所有redis服务中数据的一致性，例如请求g1在redis1中写入了421的状态，g2请求的事redis2中421的状态，可以正常获取到，但是异常是在所难免的，例如master宕机或者网络有异常这会导致很多问题发生。红锁的解决办法就是，所有进程拿锁时必须到所有redis服务的锁（把这些redis服务器都看成是平级）才算拿锁成功，如果有并发，假设存在5个服务器，比如一个进程拿到2个服务器的锁，另一个进程拿到三个服务器的锁，那就拿的多的获取锁，拿两个那个进程需要释放锁。
